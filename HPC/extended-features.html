Key Principle

The Chrome Extension is the frontend and data capture layer. All intelligence, memory, enrichment, and heavy processing lives in the backend. The extension sends data; the backend thinks.

🧠 2. Memory System

🔴 PRIORITY #1

What It Is

Every user gets persistent memory. The AI knows them — their trades, patterns, mistakes, preferences, emotional tendencies. This is what makes it feel personal, not generic.

DynamoDB Table: user_memory

PKSKDescriptionExample
USER#{userId} TRADE#{timestamp} Trade logged by user or extracted from chat LONG NQ 18200→18250, +$50, scalp, 15min hold
USER#{userId} LESSON#{date} Insight learned from trade or feedback "Low volume bounces on NQ = weak, don't trust"
USER#{userId} BEHAVIOR#{date} Emotional/behavioral pattern detected "Revenge trades after 2+ consecutive losses"
USER#{userId} PREFERENCE#{key} User preference expressed in conversation "No trades 30 min before high-impact news"
USER#{userId} FEEDBACK#{timestamp} "Mira aquí" result feedback linked to signal Signal was LONG NQ → hit TP +50pts, breakout confirmed
USER#{userId} SIGNAL#{timestamp} Trade study signal given by AI LONG NQ 18200, SL 18175, TP 18260, conf 7/10

Memory Post-Processor

Runs after every conversation ends. A Lambda calls GPT-4o-mini (~$0.001 per run) to extract structured data from the chat:

  • Did the user mention a trade? → TRADE
  • Was there a lesson learned? → LESSON
  • Did the user show emotional pattern? → BEHAVIOR
  • Did the user express a preference? → PREFERENCE
  • Did the user send result feedback? → FEEDBACK
// Lambda: memory_post_processor
async function extractMemory(userId, conversation) {
  const extraction = await openai.chat({
    model: 'gpt-4o-mini',
    messages: [{
      role: 'system',
      content: `Analyze this trading conversation. Extract:
        1. trades[] — any trade results (symbol, direction, entry, exit, pnl)
        2. lessons[] — insights learned
        3. behaviors[] — emotional patterns (revenge, tilt, overconfidence)
        4. preferences[] — stated preferences
        5. feedback[] — result screenshots/outcomes linked to prior signals
        Return JSON. Empty arrays if nothing to extract.`
    }, ...conversation]
  });

  const data = JSON.parse(extraction);

  for (const trade of data.trades) {
    await saveMemory(userId, 'TRADE', trade);
  }
  for (const lesson of data.lessons) {
    await saveMemory(userId, 'LESSON', lesson);
  }
  // ... same for behaviors, preferences, feedback
}

Memory Retrieval (injected at chat start)

async function getUserMemory(userId) {
  const trades    = await query(userId, 'TRADE#', { limit: 20, reverse: true });
  const lessons   = await query(userId, 'LESSON#', { limit: 10, reverse: true });
  const behaviors = await query(userId, 'BEHAVIOR#', { limit: 5, reverse: true });
  const prefs     = await query(userId, 'PREFERENCE#');

  return { trades, lessons, behaviors, preferences: prefs };
}

💬 3. Dynamic System Prompt

🔴 PRIORITY #1

What It Is

Every LLM request is built dynamically: Base Soul (fixed personality) + User Profile (style, instruments, risk) + User Memory (trades, lessons, behaviors) + Market Context (today's briefing, calendar) + Conversation History (last 20 messages).

async function buildSystemPrompt(userId) {
  const soul    = await getBaseSoul();       // SOUL.md content — fixed
  const user    = await getUser(userId);       // DynamoDB: profile
  const memory  = await getUserMemory(userId); // DynamoDB: trades, lessons, etc.
  const market  = await getMarketContext();     // Redis: today's briefing + calendar
  const stats   = await getUserStats(userId);  // Calculated from trade history

  return `${soul}

## Current User
- Name: ${user.name}
- Trading Style: ${user.tradingStyle}
- Instruments: ${user.instruments.join(', ')}
- Account Size: $${user.accountSize}
- Risk per Trade: ${user.riskPercent}%
- Language: ${user.language}

## What You Know About This Trader
- Win rate (30d): ${stats.winRate30d}%
- Current streak: ${stats.currentStreak}
- Trades today: ${stats.tradesToday}
- P/L today: $${stats.pnlToday}
${stats.tradesToday >= 3 && stats.pnlToday < 0
    ? '⚠️ ALERT: 3+ losing trades today. Watch for revenge trading.'
    : ''}

## Behavioral Notes
${memory.behaviors.map(b => '- ' + b.note).join('\\n')}

## Learned Lessons
${memory.lessons.map(l => '- ' + l.note).join('\\n')}

## Preferences
${memory.preferences.map(p => '- ' + p.note).join('\\n')}

## Recent Trades
${memory.trades.map(t =>
    '- ' + t.direction + ' ' + t.symbol + ' ' + t.entry + '→' + t.exit + ' ' + (t.pnl > 0 ? '+' : '') + '$' + t.pnl
  ).join('\\n')}

## Market Context Today
${market.summary}
Upcoming events: ${market.upcomingEvents}`;
}

🔄 4. Feedback Loop

🔴 PRIORITY #1

What It Is

The user sends a screenshot of what happened after a signal — "mira, se fue para arriba" — and the AI learns from the outcome. This is what makes the system get smarter over time for each specific user.

Flow

StepWhat HappensWhere
1AI gives trade study signalSaved as SIGNAL#{ts} in DynamoDB
2User sends result screenshot or text (minutes/hours later)Extension → Backend
3AI analyzes the result via vision, compares with original signalLambda + LLM
4Extracts: win/loss, what worked, what failed, lessonLLM extraction
5Saves as FEEDBACK#{ts} linked to original SIGNAL#{ts}DynamoDB
6Future analyses reference this history for similar setupsPrompt injection

Example Interaction

User: "Mira, llegó a 18,280" + screenshot

AI: "Bien ahí 🎯 El breakout de la consolidación funcionó. TP1 hit +45pts. Noto que el volumen subió justo en la ruptura — eso lo guardo como confirmación para setups similares. Tu win rate en breakouts de NQ sube a 68%."

🔀 5. Model Router

🟠 PRIORITY #2

Why

OpenAI refuses trade signals due to financial advice guardrails. Claude doesn't. Route trade analysis to Claude, cheap tasks to GPT-4o-mini, deep research to GPT-5. Saves cost and avoids refusals.

TaskModelProviderWhy
Chart analysis / Trade signals claude-sonnet-4-5 Anthropic No financial restrictions, great vision
Quick chat / Q&A gpt-4o-mini OpenAI Cheap, fast
Deep research / Complex analysis gpt-5 OpenAI Best reasoning
Memory extraction (post-chat) gpt-4o-mini OpenAI Classification task, cheap
Intent classification gpt-4o-mini OpenAI ~$0.0005 per call 
function routeModel(intent, hasImage) {
  if (intent === 'trade_signal' || intent === 'chart_analysis') {
    return { provider: 'anthropic', model: 'claude-sonnet-4-5-20250514' };
  }
  if (intent === 'deep_research') {
    return { provider: 'openai', model: 'gpt-5' };
  }
  return { provider: 'openai', model: 'gpt-4o-mini' };
}

Key Insight: TradingView IS Your Real-Time Feed

The user has TradingView open with real-time data. The extension scrapes the current price from the DOM and sends it to the backend. Zero cost for real-time prices. Backend APIs are only for enrichment (correlations, calendar, news) — those don't need to be real-time.

🎯 7. Trade Study Signals

🔴 PRIORITY #1 — exists, needs polish

Signal Output Format

📊 NQ (Nasdaq 100) — LONG Entry: 18,245 SL: 18,220 (-25 pts) TP1: 18,290 (+45 pts) ← 1.8R TP2: 18,330 (+85 pts) ← 3.4R Size: 0.02 lots (based on $50 account, 10% risk) Confidence: 7/10 Why: Support at 18,240 held 3 times, RSI bouncing from 35, DXY dropping, no news next 2h. Scalp target 15-20 min. ⚠️ If 18,220 breaks, exit. Do not move SL.

What triggers a signal

  • User clicks "🎯 Dame Entrada" button → auto-captures screenshot + DOM data
  • User types request with trade intent (semantic detection)
  • One click = full signal with SL, TP, size. Zero typing required.

What the backend does on signal request

  1. Receive: screenshot + symbol + timeframe + price + indicators (from extension)
  2. Enrich: economic calendar check, correlation check (DXY/VIX), news sentiment
  3. Load: user memory (recent trades, lessons, behavioral warnings)
  4. Calculate: position size based on user's account + risk tolerance
  5. Route to Claude Sonnet 4.5 (no financial restrictions)
  6. Return structured signal to chat
  7. Save signal as SIGNAL#{ts} for future feedback tracking

const cron = require('node-cron');

// Runs every minute
cron.schedule('* * * * *', async () => {
  const due = await getDueSchedules(); // DynamoDB query: nextRun <= now
  for (const s of due) {
    await executeTask(s);          // Run AI analysis, send reminder, etc.
    await updateNextRun(s);         // Calculate next occurrence
  }
});

async function deliver(userId, message, channel) {
  switch (channel) {
    case 'telegram':
      await sendTelegram(await getChatId(userId), message);
      break;
    case 'extension':
      await saveNotification(userId, message); // Extension polls for these
      break;
    case 'email':
      await sendEmail(userId, message);
      break;
  }
}

// Daily cron at 6 AM ET: schedule alerts for today's events
cron.schedule('0 6 * * 1-5', async () => {
  const events = await getEconomicCalendar('today');
  for (const event of events.filter(e => e.impact === 'HIGH')) {
    const alertTime = new Date(event.time - 30 * 60 * 1000); // 30 min before

📋 9. Pre-Market Briefings

🟠 PRIORITY #2

How It Works

  1. Cron runs at 8:00 AM ET weekdays
  2. Lambda generates briefing per active instrument (NQ, Gold, BTC, etc.)
  3. Stored in DynamoDB: BRIEFING#{symbol}#{date}
  4. Extension polls GET /api/briefings?since=lastSeen
  5. New briefing → badge on extension icon + bubble in chat
  6. Also pushed to Telegram for users who have it connected

Briefing is per MARKET, not per user

One NQ briefing serves all NQ traders. One Gold briefing serves all Gold traders. Each user sees only the instruments from their profile. The analysis is identical — only the delivery is personalized.

📜 14. SOUL.md Reference

Location: /HPC/SOUL.md

The SOUL.md file in this same directory contains the complete base system prompt for the HPC AI. It defines:

  • Identity — what the AI is (educational trading study companion)
  • Personality — direct, bilingual, confident, empathetic, adaptive
  • Signal format — exact structure for trade study setups
  • Memory rules — how to use and update user memory
  • Emotional awareness — when and how to flag behavioral patterns
  • Available tools — all function calling tools the AI can invoke
  • Formatting rules — markdown in extension, mobile-friendly in Telegram
  • Legal framing — educational tool, not financial advice

This file is loaded as the first section of every system prompt. User-specific context is appended dynamically by the buildSystemPrompt() function.
